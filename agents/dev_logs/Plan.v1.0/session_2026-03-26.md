## 2. CoreML / NPU Acceleration Visibility — `asr.py`

**Problem**: Service started via `./start_server.sh` showed no indication of whether CoreML (Apple Silicon ANE/GPU) was active. The print statement only existed in the CLI `main()` path, not in `load_recognizer()` itself.

**Root cause**: `app.py` calls `load_recognizer()` directly (not via `main()`), so the CoreML label printed in `main()` was never reached.

**Fix**: Added one print line inside `load_recognizer()` before building the recognizer:

```python
print(f"[ASR] 推理后端: {provider}（模型类型: {model_type}，线程数: {NUM_THREADS}）")
```

**Output on startup**:
```
[ASR] 推理后端: coreml（模型类型: sense_voice，线程数: 4）
[ASR] 模型就绪
```

**Detection logic** (`_use_coreml()`): `platform.system() == "Darwin" and platform.machine() == "arm64"` — covers all Apple Silicon including M4.

**Note**: CoreML with ONNX Runtime delegates to GPU or ANE at Apple's discretion. Use `sudo powermetrics --samplers gpu_power` to observe actual hardware utilization.

## 3. Pipeline Redesign — Raw ASR Temp File + Resumable LLM Step

**Problem**: Previous flow deleted the WAV immediately after ASR and wrote LLM-completed text directly to userdata. If LLM failed or no API key was provided, no intermediate state was preserved.

**New pipeline**:

```
Download → WAV (data/temp/)
  → ASR transcribe
  → Save raw text → data/temp/<BVID>-<title>-raw.txt
  → [If no API key] → stop here, leave raw in temp, delete WAV
  → LLM 补全 (reads raw temp file)
  → Write transcript → data/userdata/<stem>-transcript.txt
  → Delete raw temp file
  → LLM summarize
  → Write summary → data/userdata/<stem>-summary.txt
  → Delete WAV
```

**Resume logic (downloader thread)**:
- If `data/userdata/<bvid>-*-transcript.txt` AND `-summary.txt` both exist → skip entirely
- If transcript exists but no summary → skip to summary step
- If `data/temp/<bvid>-*-raw.txt` exists → **skip download + ASR**, resume from LLM step
- Otherwise → full download + transcribe

**WAV lifetime**: Kept alive until both transcript AND summary are written. Deleted on error or after full completion.

**Key design decisions**:
- Raw temp naming: `<BVID>-<safe_title>-raw.txt` (glob `<bvid>-*-raw.txt` for lookup)
- No API key path: WAV deleted after raw temp saved; no userdata output; raw remains for future retry with key
- raw temp deleted only AFTER `t_path.write_text()` succeeds (atomic-ish safety)

**Files changed**: `app.py` — `_processing_worker()`, both `downloader()` and `processor()` closures.

**Commits**: `7133435` (pipeline), `7af5ef8` (WAV lifetime fix)

## 4. Cookie Field UI Improvement — `config.html`

**Problem**: The cookie input had a minimal placeholder `SESSDATA=xxx; bili_jct=yyy; ...` with no explanation of which fields are needed, their purpose, or how to obtain them.

**Investigation**: Bilibili-Evolved's `download.js` uses `getJsonWithCredentials` — a browser extension abstraction that taps the browser's own cookie store. It does not hardcode cookie names. The required cookies were determined from API behavior and community knowledge.

**Note**: The `SocialSisterYi/bilibili-API-collect` documentation repo was permanently taken down in January 2026 following a cease-and-desist from Bilibili's legal team.

**Cookies identified**:

| Cookie | Format | Priority |
|---|---|---|
| `SESSDATA` | URL-encoded token string | 必须 |
| `bili_jct` | 32-char hex | 必须 |
| `DedeUserID` | integer | 必须 |
| `DedeUserID__ckMd5` | 32-char hex | 推荐 |
| `buvid3` | UUID-like + `infoc` suffix | 推荐 |
| `buvid4` | UUID-like + `infoc` suffix | 推荐 |
| `sid` | 8-char alphanumeric | 可选 |

**UI changes**:
- `config.html`: expanded label, realistic placeholder showing all 7 fields, added `.field-hint` paragraph with step-by-step instructions (F12 → Application → Cookies → bilibili.com)
- `static/style.css`: added `.field-hint` and `.field-hint code` styles
